## 三、什么是外部库 External Library

### 1. 外部库的定义

对应视频时间：`00:00:05`

课程一开始给出的核心定义是：

> 外部库就是项目之外的代码。

也就是说：
- 你自己项目里的 `cart.js`、`products.js`、`checkout.js` 不是外部库
- 别人写好并发布到网上，能被你直接拿来用的代码，才叫外部库

比如：
- 日期处理库
- 动画库
- 图表库
- 工具函数库

---

### 2. 为什么要用外部库

对应视频时间：`00:14:18`

课程强调了一个很重要的开发习惯：

> 当你要做比较复杂的事情时，先找有没有外部库，而不是立刻自己从零实现。

原因有三点：

1. 节省时间
2. 减少 bug
3. 站在已有成熟方案上开发

这一课里的典型例子就是“日期处理”。

如果你自己处理日期，会很快碰到这些问题：
- 加几天后的日期怎么计算
- 月份跨月怎么处理
- 星期几怎么显示
- 周末要不要跳过
- 格式化输出怎么写

这些问题都不适合初学者在项目里手搓。

## 四、从互联网加载外部库

### 1. 最简单的加载方式：`script src="网址"`

对应视频时间：`00:00:44`

课程先演示了最直接的方式：

```html
<script src="https://example.com/some-library.js"></script>
```

浏览器会：

1. 去这个网址下载 JavaScript 代码
2. 把它像普通脚本一样加载到当前页面
3. 让你可以在项目里直接使用库暴露出来的变量或函数

---

### 2. 为什么这种方式能工作

对应视频时间：`00:02:48`

因为这个远程脚本一旦被加载，就相当于它的代码也在当前页面执行了。

如果这个库里有：

```js
function hello() {
  console.log('hello');
}
```

那么你在自己的代码里就能直接写：

```js
hello();
```

---

### 3. 这种方式的问题

对应视频时间：`00:15:01`

课程后面指出，普通 `script` 标签加载外部库有一个明显问题：

> 它会直接把代码运行在页面环境里，容易造成命名冲突。

比如：
- 库里有个 `hello`
- 你自己代码里也有个 `hello`

那就可能冲突。

这和上一课讲模块时提到的全局污染是同一个问题。

## 五、压缩代码 Minification

### 1. 什么是压缩

对应视频时间：`00:06:14`

课程在展示 Day.js 源码时提到，很多外部库代码看起来非常难读：
- 空格被删掉
- 变量名很短
- 一整行很长

这是因为库作者通常会把代码做“压缩”。

英文叫：
- `minification`
- `minify the code`

---

### 2. 为什么要压缩

对应视频时间：`00:06:25`

因为外部库通常是通过网络下载的，文件越小：
- 下载越快
- 页面加载越快

所以给浏览器用的版本通常是压缩后的。

补充理解：
- 给人读的代码：更注重可读性
- 给浏览器下载的代码：更注重体积和性能

## 七、Day.js 解决日期问题的三步

课程把日期处理拆成了三个任务。

### 1. 获取当前日期

对应视频时间：`00:09:19`

基本写法：

```js
const today = dayjs();
```

这会得到一个表示“当前日期和时间”的对象。

你可以把它理解成：

> Day.js 帮我们创建了一个更方便操作的日期对象。

---

### 2. 做日期计算：`add()`

对应视频时间：`00:10:20`

课程演示：

```js
const deliveryDate = today.add(7, 'days');
```

含义：
- 在 `today` 的基础上
- 加 `7`
- 单位是 `'days'`

也就是“7 天后”。

常见形式：

```js
today.add(1, 'day');
today.add(3, 'days');
today.add(1, 'month');
```

---

### 3. 格式化日期：`format()`

对应视频时间：`00:12:01`

课程演示：

```js
deliveryDate.format('dddd, MMMM D');
```

作用：
- 把日期对象转成更适合显示的字符串

像这种格式会输出类似：

```js
'Tuesday, June 18'
```

其中：
- `dddd` 表示星期几全名
- `MMMM` 表示月份全名
- `D` 表示几号

补充理解：

程序里通常会区分两种东西：
- 用来计算的日期对象
- 用来显示的日期字符串

不要把这两者混在一起。

## 九、外部库和 JavaScript 模块一起使用

### 1. 为什么课程要从 `script` 方式升级到模块方式

对应视频时间：`00:15:01`

原因还是上一课的核心问题：
- 避免命名冲突
- 避免把所有东西丢到全局里
- 让依赖关系更明确

所以课程说，更好的做法是：

> 用 JavaScript 模块来加载外部库。

---

### 2. 什么是 ESM 版本

对应视频时间：`00:15:40`

课程提到：

> 要和模块一起使用外部库，就需要库的 ESM 版本。

ESM 是：
- `ECMAScript Modules`

也就是：
- JavaScript 模块标准版本

简单理解：

> ESM 版本就是“支持 `import` / `export` 的库版本”。

---

### 3. 从 URL 直接导入模块

对应视频时间：`00:16:51`

课程演示了这种写法：

```js
import { hello } from 'https://example.com/hello.js';
```

或者：

```js
import dayjs from 'https://example.com/dayjs.js';
```

这和导入本地文件的思路一样，只不过：
- 本地文件导入：`./`、`../`
- 远程模块导入：直接用 URL

---

### 4. 导入远程模块时要注意什么

补充知识点：

1. 当前脚本本身必须是模块脚本

```html
<script type="module" src="scripts/checkout.js"></script>
```

2. 远程资源要支持模块加载
3. 某些环境里会受 CORS 或本地文件协议限制
4. 并不是每个库都提供可直接浏览器导入的 ESM URL

课程里能直接导入，是因为提供了专门的教学 URL。

## 十、默认导出和命名导出

### 1. 命名导出 `named export`

对应视频时间：`00:21:16`

前几课你主要看到的是：

```js
export function formatCurrency(priceCents) {
  return (priceCents / 100).toFixed(2);
}
```

导入时：

```js
import { formatCurrency } from '../utils/money.js';
```

特点：
- 要用大括号
- 一个文件可以导出多个命名内容

---

### 2. 默认导出 `default export`

对应视频时间：`00:19:47`

课程这节新讲了另一种语法：

```js
export default formatCurrency;
```

导入时：

```js
import formatCurrency from '../utils/money.js';
```

特点：
- 不需要大括号
- 每个文件只能有一个默认导出

---

### 3. 什么时候适合默认导出

对应视频时间：`00:20:20`

课程里的结论是：

> 当文件只想导出一个东西时，默认导出会更简洁。

比如：
- 一个文件只导出一个主函数
- 一个文件只导出一个类
- 一个外部库只暴露一个主要入口

像 Day.js 就是这种情况。

---

### 4. 什么时候适合命名导出

命名导出更适合：
- 一个文件导出多个工具函数
- 想让导入名保持清晰一致
- 想让自动提示和重构更明确

比如：

```js
export function getProduct() {}
export function getProducts() {}
export function saveToStorage() {}
```

---

### 5. 两者最核心的区别

快速记忆：

- `named export`：导出多个，导入时用 `{ }`
- `default export`：默认只有一个，导入时不用 `{ }`

## 十一、配送选项的数据设计

### 1. 先保存数据，再生成页面

对应视频时间：`00:22:22`

课程继续沿用前面几课的三步流程：

1. 保存数据
2. 生成 HTML
3. 做交互

这说明页面开发的主线仍然是“数据驱动界面”。

---

### 2. 为什么不把完整配送信息直接存进购物车

对应视频时间：`00:23:20`

课程明确指出：

如果每个购物车项都保存完整配送信息，比如：
- 配送天数
- 运费价格

这些数据会在购物车数组里重复很多次。

于是出现两个问题：

1. 冗余
2. 修改困难

这就是课程引出“规范化数据”的原因。

---

### 3. 什么是规范化数据 `normalize data`

对应视频时间：`00:23:20`

规范化的核心思想是：

> 把重复的数据拆出去单独保存，只在当前对象里保留引用它的 ID。

所以：
- 完整配送选项单独保存在 `deliveryOptions.js`
- 购物车项只保存 `deliveryOptionId`

例如：

```js
const deliveryOptions = [
  {
    id: '1',
    deliveryDays: 7,
    priceCents: 0
  },
  {
    id: '2',
    deliveryDays: 3,
    priceCents: 499
  }
];
```

购物车里只写：

```js
{
  productId: '...',
  quantity: 2,
  deliveryOptionId: '1'
}
```

---

### 4. 为什么这种设计更好

因为这样可以：
- 减少重复
- 修改更集中
- 更接近真实数据库设计思路

这和前面课程里：
- 购物车只保存 `productId`
- 商品详情单独保存在 `products.js`

本质上是一样的。

## 十二、创建 `deliveryOptions.js`

### 1. 配送选项本质上是“对象数组”

对应视频时间：`00:23:35`

课程新建 `deliveryOptions.js`，里面保存多个配送选项。

每个配送选项至少有三个属性：

```js
{
  id: '1',
  deliveryDays: 7,
  priceCents: 0
}
```

含义：
- `id`：唯一标识
- `deliveryDays`：几天送达
- `priceCents`：运费，单位是分

---

### 2. 金额为什么继续用分 `cents`

对应视频时间：`00:25:15`

这延续了前面 Amazon 项目的金额处理规则：

> 计算钱时，尽量用整数分，不直接用小数美元。

原因：
- 浮点数计算可能有精度问题
- 整数更稳定

例如：
- `$4.99` 存成 `499`
- `$9.99` 存成 `999`

### 2. 为什么只保存 ID

> 当前对象只保存“引用关系”，真正的完整信息到专门的数据表里查。

这是一种非常重要的前端数据建模思路。

## 十四、计算配送日期

### 1. 核心思路

对应视频时间：`00:29:10`

对于每个配送选项，页面要显示一个具体日期，比如：
- Tuesday, June 18

这个日期不是写死的，而是：

> 今天 + 配送天数

所以流程是：

1. 取今天日期 `dayjs()`
2. 根据 `deliveryDays` 往后加
3. 格式化成字符串显示

---

### 2. 课程中处理周末的逻辑

对应视频时间：`00:30:35`

课程不仅仅是“直接加几天”，还加入了：

> 跳过周末

这意味着“配送天数”指的是工作日，而不是自然日。

典型思路是：

1. 准备一个日期变量
2. 循环若干次
3. 每次先往后加 1 天
4. 如果是周六/周日，就不计入配送天数
5. 只有工作日才累计一天

这类逻辑非常适合：
- 循环
- 条件判断
- 日期库

---

### 3. 为什么要用库来处理这个问题

因为日期相关的边界很多：
- 跨月
- 跨年
- 不同月份天数不同
- 周末判定

如果不用库，代码会很容易变复杂。

---

### 4. `dayjs().add()` 的一个重要特点

对应视频时间：`00:31:40`

这一点课程虽然没有用“不可变”这个术语大讲，但你应该补充理解：

> `add()` 通常返回的是一个新的日期对象，而不是直接改原对象。

比如：

```js
const today = dayjs();
const nextDay = today.add(1, 'day');
```

通常表示：
- `today` 还是原来的日期
- `nextDay` 是新日期

这类设计能减少很多意外修改问题。

## 十五、生成配送选项 HTML

### 1. 一条购物车商品对应多条配送选项

对应视频时间：`00:36:00`

这里的界面结构是：
- 左边是一条购物车商品
- 右侧下面有多个配送方式可选

所以代码一般是：

1. 外层循环购物车项
2. 内层循环配送选项

这属于“嵌套渲染”的典型场景。

---

### 2. 模板字符串的作用

对应视频时间：`00:36:00`

课程仍然是用模板字符串生成 HTML：

```js
deliveryOptionsHTML += `
  <div class="delivery-option">
    ...
  </div>
`;
```

模板字符串在这里非常适合：
- 多行 HTML
- 插入变量
- 拼接 class / 文本 / 属性

### 3. 动态 `checked` 属性

对应视频时间：`00:41:50`

课程用了三元运算符来决定某个单选框是否应该选中。

思路是：

```js
const isChecked = option.id === cartItem.deliveryOptionId;
```

如果相等，就在 HTML 里插入：

```html
checked
```

否则插入空字符串。

这是一个很典型的“根据数据生成属性”的做法。

### 4. 顶部日期要根据当前选中的配送方式变化

对应视频时间：`00:42:43`

课程后面把顶部显示的配送日期，也改成了根据当前 `deliveryOptionId` 动态计算。

流程是：

1. 从购物车项读出 `deliveryOptionId`
2. 找到完整配送对象
3. 取出 `deliveryDays`
4. 计算并格式化日期
5. 插入到 HTML

这说明：

## 十六、更新配送选项时重新生成 HTML

### 1. 为什么不直接手动改一小块 DOM

对应视频时间：`01:00:24`

课程在这里引出了后半节最核心的思想：

> 更新数据，然后重新生成 HTML。

而不是：
- 这里手动改一个日期
- 那里手动改一个价格
- 再到别处手动改一个勾选状态

因为这样很容易出现页面局部不同步。

---

### 2. 这种方式的好处

如果你遵循：

1. 数据是唯一真实来源
2. 页面只是数据的显示结果

那么更新流程就会很稳定：

1. 改数据
2. 重新渲染

这是后面很多框架思想的基础。

## 十七、MVC 设计模式

### 1. MVC 是什么

对应视频时间：`01:00:37`

MVC 是：

- `Model`
- `View`
- `Controller`

是一种组织代码的设计模式。

课程里把它介绍为：

> 一种把代码分成三部分，并让它们协作的方式。

---

### 2. Model：模型

对应视频时间：`01:00:48`

模型负责：
- 保存数据
- 管理数据

在这个项目里，典型 Model 包括：
- `cart.js`
- `products.js`
- `deliveryOptions.js`

这些文件都属于“数据层”。

---

### 3. View：视图

对应视频时间：`01:01:15`

视图负责：
- 读取数据
- 生成 HTML
- 把数据展示到页面上

在这个项目里：
- `orderSummary.js` 里生成左侧订单列表的代码
- `paymentSummary.js` 里生成右侧价格区域的代码

都属于 View。

---

### 4. Controller：控制器

对应视频时间：`01:01:36`

控制器负责：
- 响应用户操作
- 决定要更新什么数据
- 决定何时重新渲染

比如：
- 点击删除按钮
- 切换配送选项

这些事件监听器就是 Controller。

---

### 5. 这三者是怎么协作的

对应视频时间：`01:02:01`

课程中的完整循环是：

1. 用 Model 数据生成 View
2. 用户和 View 交互
3. Controller 响应交互并更新 Model
4. 用更新后的 Model 重新生成 View

这是本课最关键的工程思维。

---

### 6. 为什么课程强调 MVC

对应视频时间：`01:03:03`

课程明确说：

> 不直接到处手动改 DOM，而是更新数据后重新生成 HTML，能确保页面始终和数据一致。

这句话一定要记住。

它是本课里最有“架构味道”的内容。

## 十八、拆分 checkout 页面代码

### 1. 为什么要拆文件

对应视频时间：`01:04:40`

课程指出 `checkout.js` 代码已经很多了，所以要拆。

拆分的理由很实际：
- 文件太大不容易读
- 不同区域逻辑混在一起
- 后续维护困难

---

### 2. 拆成哪两个文件

对应视频时间：`01:04:56`

课程把 checkout 页面拆成：

- `checkout/orderSummary.js`
- `checkout/paymentSummary.js`

职责分别是：
- 左侧订单区域
- 右侧付款区域

这是很典型的“按页面区域拆分”。

---

### 3. 入口文件的作用

对应视频时间：`01:05:35`

拆分后，`checkout.js` 不再承载所有逻辑，而更像一个入口文件：
- 导入两个模块
- 调用它们的渲染函数

这和上一课的 `entry point` 思想是连在一起的。

---

### 4. 文件移动后，相对路径怎么改

对应视频时间：`01:06:55`

这一段非常适合初学者重点掌握。

当 `orderSummary.js` 被移动到 `checkout/` 子文件夹后，原来的导入路径可能失效。

比如原来：

```js
import { cart } from '../data/cart.js';
```

如果文件层级变了，路径就要跟着改。

快速记忆：
- `./` 当前文件夹
- `../` 上一级文件夹
- `../../` 上两级文件夹

判断路径时，不是看 HTML 文件，而是：

> 从当前这个 JS 文件所在的位置开始算。

这是相对路径最容易搞错的地方。

## 二十一、计算 payment summary

## 二十五、本课应该补充理解的相关知识点

### 1. CDN 与远程 URL 加载

这节课虽然没重点讲“CDN”这个词，但你看到的那种 URL 远程加载，本质上和 CDN 分发思路有关。

简单理解：

> 别人把库文件放到一个可公开访问的网址上，你的浏览器直接下载使用。

实际开发里常见方式有两种：
- `<script src="...">`
- `import ... from 'https://...'`

---

### 2. 浏览器环境和打包器环境的区别

课程这里是在浏览器里直接导入 URL。

但真实项目里更常见的是：
- `npm install`
- 然后在源码里 `import dayjs from 'dayjs'`
- 最后由打包工具处理

所以这节课更像是在帮你先理解“模块和库”的底层关系。

---

### 3. 数据建模

这一课其实已经在训练你做简单的数据建模：
- 商品数据单独存
- 配送选项单独存
- 购物车只保留引用字段

这和数据库表设计思想非常接近。

---

### 4. 重新渲染不是性能问题的第一来源

初学者容易担心：

> 每次都重新生成 HTML，会不会很慢？

在这个课程规模里，这完全不是问题。

更重要的是先建立正确的更新模型：
- 保证数据和页面一致
- 代码结构清晰

等后面数据量变大，再考虑更细粒度的更新策略。

## 二十六、常见错误整理

### 1. 把外部库当成浏览器内置

错误理解：
- 以为 `dayjs()` 天生就能用

正确理解：
- 必须先加载或导入这个库

---

### 2. `default export` 和 `named export` 混用

常见错误：

```js
import { dayjs } from '...';
```

如果对方导出的是默认导出，就不该加大括号。

---

### 3. 相对路径从错的地方开始算

最常见错误之一：
- 从 HTML 文件位置算
- 或从项目根目录凭感觉算

正确方法：

> 永远从当前 JS 文件所在位置开始算相对路径。

---

### 4. 购物车里保存了重复的完整配送数据

这样会导致：
- 数据冗余
- 修改麻烦
- 后续容易不同步

更好的做法是只保存 `deliveryOptionId`。

---

### 5. 更新数据后忘了重新渲染

比如：
- 删除商品后数据变了，但右侧价格没更新
- 切换配送方式后顶部日期没更新

这通常就是因为：

```js
renderPaymentSummary();
```

或对应渲染函数没有重新调用。

---

### 6. 直接手动到处改 DOM，导致状态不一致

例如：
- 改了顶部日期
- 却忘了改右侧运费
- 又忘了改单选框状态

这正是 MVC 想避免的问题。
