## 三、为什么需要模块

### 1. 普通 `script` 标签的问题

对应视频时间：`00:00:24`

课程一开始指出，在 `amazon.html` 里连续加载多个脚本：

```html
<script src="data/cart.js"></script>
<script src="data/products.js"></script>
<script src="scripts/amazon.js"></script>
```

这种方式的一个大问题是：

> 所有文件里的变量好像都被放进了同一个大空间里。

所以如果两个文件里都写：

```js
const cart = [];
```

就会冲突。

---

### 2. 命名冲突 naming conflict

对应视频时间：`00:01:00`

课程演示了：
- `cart.js` 里已经有 `cart`
- `amazon.js` 再声明 `cart`
- 页面直接报错

这就叫命名冲突。

它的本质是：

> 两个变量处在同一个可见范围里，并且名字相同。

---

### 3. 模块解决什么问题

对应视频时间：`00:02:48`

课程给出的核心结论是：

> 模块会把变量限制在文件内部。

也就是说：
- 文件 A 里的变量，不会自动跑到文件 B 里
- 除非你明确导出和导入

这会让多文件项目更可控。

---

## 四、模块的基本思路

### 1. 模块不是“新语言”，而是新组织方式

对应视频时间：`00:03:23`

课程里其实表达了一个非常重要的意思：

> 模块不是让你写完全不同的 JavaScript，而是让你更有组织地使用多文件。

所以你可以把模块理解成：

> 给多文件项目加上一套“明确的数据流规则”。

---

### 2. 创建模块的基本思路

对应视频时间：`00:03:23`

课程这里的直觉版说法可以整理成：

1. 把代码写到独立文件里
2. 不再靠普通 `script` 标签把所有文件直接塞到全局里
3. 需要什么就显式导出 / 导入

---

## 五、从模块中取变量的三步

课程反复强调了这套流程。

### 1. 第一步：`type="module"`

对应视频时间：`00:06:24`

```html
<script type="module" src="scripts/amazon.js"></script>
```

作用：

> 告诉浏览器，这个脚本按模块方式运行。

如果没有这个属性，`import` / `export` 就不能正常按模块语法工作。

---

### 2. 第二步：`export`

对应视频时间：`00:06:56`

在 `cart.js` 里：

```js
export let cart = [];
```

作用：

> 明确告诉其他文件，这个变量允许被外部使用。

---

### 3. 第三步：`import`

对应视频时间：`00:07:29`

在 `amazon.js` 里：

```js
import { cart } from '../data/cart.js';
```

作用：

> 从另一个模块文件中取出你需要的变量。

---

## 六、模块中的路径写法

### 1. `../` 表示上一级文件夹

对应视频时间：`00:08:03`

在 `scripts/amazon.js` 中导入 `data/cart.js`：

```js
import { cart } from '../data/cart.js';
```

为什么是 `../`？

因为：
- 当前文件在 `scripts/`
- 想去 `data/`
- 要先退回上一层，再进入 `data`

---

### 2. `./` 表示当前文件夹

对应视频时间：`00:54:56`

例如在 `scripts/checkout.js` 中导入 `scripts/utils/money.js`：

```js
import { formatCurrency } from './utils/money.js';
```

因为：
- 当前文件已经在 `scripts/`
- `utils/` 就在当前目录下面

所以直接 `./utils/...`

---

### 3. 初学者记忆法

- `./`：从当前目录出发
- `../`：先退一级再找

这是模块里非常高频的知识点。

---

## 七、模块的注意事项

### 1. `import` 要放在文件顶部

对应视频时间：`00:10:08`

课程明确提到：

> 模块里的导入语句应该放在文件顶部。

这样更清晰，也符合常见工程规范。

---

### 2. 模块需要用 Live Server 运行

对应视频时间：`00:10:13`

课程演示了：
- 直接双击 HTML 文件打开，模块不工作
- 用 Live Server 打开，模块正常工作

所以这节课之后，如果你在本地测试模块代码，默认要：

> 用本地开发服务器运行页面，而不是直接打开文件。

---

## 八、模块的优势

### 1. 避免命名冲突

对应视频时间：`00:11:13`

模块让变量默认待在自己的文件里，不会随意污染别的文件。

所以多文件项目里，这会极大减少冲突。

---

### 2. `import as` 重命名

对应视频时间：`00:11:40`

课程展示了：

```js
import { cart as myCart } from '../data/cart.js';
```

作用：
- 导入时顺便换个名字
- 用来避免冲突或提高可读性

---

### 3. 不用再手动担心脚本顺序

对应视频时间：`00:12:35`

在老的多 `script` 标签模式下：
- 你必须先加载依赖文件
- 再加载使用它的文件

模块化后：
- 入口文件导入它需要的内容
- 结构更自然

课程把这点作为模块的第二大优势。

---

## 九、入口文件 Entry Point

### 1. 什么是入口文件

对应视频时间：`00:17:15`

课程说：

> 我们仍然需要一个脚本文件真正被 HTML 加载，这个文件就是入口点。

比如：
- 首页入口点是 `amazon.js`
- checkout 页入口点是 `checkout.js`

它们的任务是：
- 从别的模块导入内容
- 组织并运行页面逻辑

---

### 2. 为什么不是所有文件都直接放进 HTML

因为那样又回到了“所有脚本都直接全局执行”的旧模式。

模块的目标是：
- 只让入口文件直接跑
- 其他文件作为依赖被导入

这说明：

### 3. 把 `addToCart` 移到 `cart.js`

对应视频时间：`00:22:12`

课程进一步做了更合理的组织：
- `addToCart` 管购物车数据
- 所以它应该属于 `cart.js`

> 与购物车有关的逻辑，尽量放进 `cart.js`。

### 4. 一个 `import` 里导入多个内容

对应视频时间：`00:24:28`

```js

```

说明：
- 同一个文件可以同时导出多个东西
- 导入时可以一起取

### 5. `import * as ...` 的概念

对应视频时间：`00:25:09`

课程提到还有一种写法：

```js

```

可以理解成：

> 把这个模块导出的所有内容都打包成一个对象。

然后通过：

```js
cartModule.cart
cartModule.addToCart()
```

去访问。

当前阶段先知道这是一种“整包导入”的方式即可。

## 十三、checkout 页面的数据来源

### 1. 订单摘要先从购物车数据出发

对应视频时间：`00:31:00`

checkout 左侧的 order summary，本质上显示的是：
- 购物车里有哪些商品
- 每种商品买了多少

所以第一层数据来源是：

```js
cart
```

---

### 2. 为什么购物车里不直接存全部商品信息

对应视频时间：`00:34:40`

课程这里讲了一个非常重要的工程点：

购物车里只存：
- `productId`
- `quantity`

不直接重复存：
- image
- name
- price

因为这些信息已经在产品数据里有了。

## 十四、去重 / 规范化数据

### 1. 本课想表达的思想

对应视频时间：`00:35:14`

课程把这种做法描述为：
- deduplicate data
- normalize data

可以简单理解成：

> 不要在多个地方重复存同一份完整信息。

购物车只保留最关键的关联信息 `productId`，其余信息去产品表里查。

---

### 2. 为什么这样更好

因为如果重复存：
- 商品改名后要改多处
- 商品价格更新后要改多处
- 更容易产生数据不一致

规范化后：
- 商品详情只维护一份
- 购物车只维护关联关系和数量

这是非常重要的数据设计思路。

## 十五、在 checkout 中通过 `productId` 查完整商品

### 1. 基本流程

对应视频时间：`00:39:33`

思路是：

1. 先从 `cartItem` 里取出 `productId`
2. 再遍历 `products`
3. 找到 `product.id === productId` 的那项
4. 保存为 `matchingProduct`

---

### 2. 典型结构

```js
const productId = cartItem.productId;

let matchingProduct;

products.forEach((product) => {
  if (product.id === productId) {
    matchingProduct = product;
  }
});
```

---

### 3. 为什么这一步重要

它把两份数据关联起来了：
- 购物车数据
- 产品数据

这就是项目里常见的数据关联思维。

---

## 十六、生成 checkout 页的 order summary HTML

### 1. 基本思路

对应视频时间：`00:45:00`

流程和首页类似：

1. 遍历 `cart`
2. 对每个 `cartItem` 找到匹配产品
3. 生成一段 HTML
4. 把所有 HTML 拼起来
5. 插入页面

---

### 2. 和首页相比有什么不同

首页：
- 直接遍历 `products`

checkout：
- 先遍历 `cart`
- 再通过 `productId` 查 `products`

所以 checkout 比首页多了一层关联查询。

## 十七、把金额格式化工具抽到 `utils/money.js`

### 4. 工具模块复用的价值

对应视频时间：`00:57:40`

## 十八、单选框 radio selector 的 `name` 规则

### 1. 本课问题

对应视频时间：`00:58:48`

课程发现 checkout 页的配送选项会互相影响：
- 选上面一组
- 下面一组会被取消

---

### 2. 原因

因为这些 radio input 使用了相同的 `name`。

HTML 规则是：

> 同一个 `name` 的 radio 视为同一组，只能选一个。

---

### 3. 解决方法

对应视频时间：`01:00:56`

给每个商品的配送选项生成不同的 `name`，例如把商品 `id` 拼进去：

```html
name="delivery-option-${matchingProduct.id}"
```

这样：
- 一个商品的 3 个选项是一组
- 另一个商品的 3 个选项是另一组

---

这个拆分非常合理，因为它再次体现：

## 二十、把 `removeFromCart` 放到 `cart.js`

### 1. 为什么应该放进去

对应视频时间：`01:09:00`

课程强调：

> `cart.js` 应该包含所有与购物车相关的代码。

所以“从购物车删除商品”当然属于 `cart.js` 的职责。

---

### 2. 删除思路

课程这里用了“新建一个数组”的方式：

1. 创建 `newCart`
2. 遍历原 `cart`
3. 如果当前项不是要删除的那个，就加入 `newCart`
4. 最后让 `cart = newCart`

这是一种很清楚的删除思路。

---

### 3. 为什么 `cart` 这里要用 `let`

对应视频时间：`01:10:56`

因为最后做了：

```js
cart = newCart;
```

这属于重新赋值。

所以 `cart` 不能再是 `const`，而要改成 `let`。

## 二十一、`Element.remove()` 从页面移除元素

### 1. 基本做法

对应视频时间：`01:18:20`

课程先用特殊 class + `productId` 找到要删的那块容器：

```js
const container = document.querySelector(`.js-cart-item-container-${productId}`);
```

然后：

```js
container.remove();
```

---

### 2. 这一步说明了什么

说明 DOM 元素不只是能：
- 查找
- 改文本
- 改 HTML

还可以直接从页面里删除。

## 二十二、本地存储 `localStorage`

### 1. 为什么要用本地存储

对应视频时间：`01:20:30`

课程指出一个问题：
- 购物车如果只存在变量里
- 刷新页面或切换页面后就会丢失

所以需要一种：

> 页面刷新后仍能保留数据的存储方式。

这就是 `localStorage`。

---

### 2. 保存数据：`setItem`

对应视频时间：`01:21:40`

```js
localStorage.setItem('cart', JSON.stringify(cart));
```

含义：
- 第一个参数：保存时使用的键名
- 第二个参数：保存的字符串数据

---

### 3. 为什么要 `JSON.stringify`

因为：

> `localStorage` 只能保存字符串。

而 `cart` 是数组 / 对象结构，所以必须先转成 JSON 字符串。

---

### 4. 读取数据：`getItem`

对应视频时间：`01:23:20`

```js
const cart = JSON.parse(localStorage.getItem('cart'));
```

`getItem` 读出来的还是字符串，需要再转回数组 / 对象。

---

### 5. 为什么要 `JSON.parse`

因为你保存进去的是字符串化后的 JSON。

读取时要把它还原成 JavaScript 数据结构。

---

### 6. 空值 `null` 的情况

对应视频时间：`01:24:08`

第一次使用网站时，本地存储里可能还没有 `cart`。

这时候：

```js
localStorage.getItem('cart')
```

会得到 `null`。

所以课程用了：

```js
if (!cart) {
  cart = [默认值];
}
```

给它兜底。

## 二十三、本课隐含但非常重要的工程思想

### 1. 文件职责要清楚

这一课非常强调：
- `cart.js` 管购物车
- `products.js` 管产品数据
- `money.js` 管金额格式化
- `amazon.js` / `checkout.js` 管页面入口逻辑

这就是模块化真正的意义。

---

### 2. 页面只是数据的表现层

删除商品时：
- 真正重要的是先改 `cart`
- DOM 只是把变化反映出来

如果只删页面不删数据，刷新就会回来。

---

### 3. 工具函数应该复用，而不是复制

`formatCurrency` 的抽取非常典型。

这是从“会写功能”走向“会维护代码”的重要一步。

---

### 4. 数据规范化能减少重复

购物车只存 `productId` + `quantity`，不是巧合，而是一种很常见的数据设计思路。

以后你做更大项目时会反复遇到。

---

## 二十四、常见错误

### 1. 忘记 `type="module"`

没有它，模块语法可能无法正常工作。

---

### 2. 路径写错

最常见的是：
- 该用 `./` 的地方写成 `../`
- 该退一级却没退

---

### 3. 把 `import` 写在文件中间

虽然有时工具链会处理，但在本课环境里应该老老实实放顶部。

---

### 4. 同一段金额格式化逻辑复制多次

这会制造重复代码，后面维护很痛苦。

---

### 5. radio 的 `name` 没按商品区分

这会导致多个商品的选项互相影响。

---

### 6. 删除时只改 DOM，不改数据

刷新页面后商品又会回来。

---

### 7. 存本地存储时直接存数组对象

错误：

```js
localStorage.setItem('cart', cart);
```

应该先 `JSON.stringify(cart)`。

---

### 8. 读取本地存储后忘记 `JSON.parse`

这样读出来只是字符串，不能正常按数组处理。

## 二十六、复习版总结

这一课最核心的内容可以压缩成下面几句话：

- 模块是组织多文件 JavaScript 代码的更好方式。
- 模块默认把变量限制在文件内部，避免随意污染其他文件。
- 使用模块的基本流程是：`type="module"`、`export`、`import`。
- 多文件项目中，路径写法很关键，`./` 是当前目录，`../` 是上一级目录。
- 入口文件负责真正启动页面逻辑，其他文件通过模块被导入。
- checkout 页面通过购物车里的 `productId` 去产品数据里查完整商品信息，这是一种规范化数据思路。
- 工具函数应该抽到共享模块里，避免重复代码。
- 删除商品时要同时更新数据和 DOM。
- `localStorage` 可以让购物车在刷新页面和跨页面时仍然保留。
- `localStorage` 只能保存字符串，所以需要 `JSON.stringify` 和 `JSON.parse`。
